## Задание 1. СУБД

Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для каждой предметной области.

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?
1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков. СУБД должна гарантировать целостность и чёткую структуру данных.

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы?

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? СУБД должны быть гибкими и быстрыми.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это реализовать?

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать со связями.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД логистов?

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

### Решение 

1.1. Для решения задач по бюджетированию проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков лучше всего на мой взгляд подойдет Реляционная СУБД (PostgreSQL).

Потму что 

   1 Финансы не терпят ошибок. Реляционные БД гарантируют целостность (нельзя положить полмиллиона и потерять вторую половину)
   2 Чёткая структура: дебет/кредит не плавают, а строго в таблицах
   3 Сложные отчёты и прогнозы легко делать через JOIN и агрегатные функции

1.1. Ответ: Использовать аппаратное ускорение через Intel QAT или AES-NI (API: OpenSSL engine, af_alg kernel API)
     Пояснение: Хеширование в финансах нужно для проверки целостности данных. Процессор современный умеет это делать быстро, но нужно задействовать спец-инструкции.    Это как нанять 10 кассиров вместо одного.

1.2 - Лендинги + CRM

Ответ:
Для лендингов: Документная СУБД (MongoDB)
Для CRM: Тоже документная или реляционная (PostgreSQL)
Почему:
Лендинги постоянно меняются: сегодня одно поле, завтра другое. В реляционной БД пришлось бы менять структуру таблицы (миграции) - это больно
Документные БД позволяют хранить разные наборы полей в одной коллекции.

1.2. Можно ли одной СУБД?*
Ответ: Да, PostgreSQL с JSONB (гибридный подход)

1.3 - Корпоративные нормы и правила
Ответ: Иерархическая/документная СУБД (MongoDB) или просто файловая система + поиск (Elasticsearch)

Почему:

Структура компании - это дерево (директор → департаменты → отделы → сотрудники)
Документные БД отлично хранят вложенные структуры
Простота: можно даже на Markdown файлах в Git сделать

1.3. Можно ли использовать существующую?*
Ответ: Да, можно добавить как схему в PostgreSQL или как отдельную коллекцию в MongoDB

Лучшая реализация: Оставить отдельно (MongoDB или Elasticsearch), потому что:

Поиск по документам будет быстрее (полнотекстовый поиск)

Нагрузка на финансовую БД не вырастет

1.4 - Логистика (маршруты)
Ответ: Графовая СУБД (Neo4j, ArangoDB)

Почему:

Маршруты - это чистая графовая задача (точки А→Б→В со связями)

В реляционной БД пришлось бы делать JOIN на JOIN (медленно)

Графовые БД хранят связи как первоклассные объекты

1.4. Отдел закупок - отдельно или вместе?**
Ответ: Лучше отдельная, но в связке (микросервисная архитектура)

Как связать:

У логистов: Neo4j (маршруты, курьеры)

У закупок: PostgreSQL (заказы, поставщики, счета)

Связь через API: логисты говорят "мне нужно 5 тонн цемента на стройку X", закупки отвечают "заказали у поставщика Y, привезёт через 2 дня"

*Задание 1.5 - Можно ли всё одной СУБД?**
Ответ: Можно, но не нужно. Если очень хочется - PostgreSQL справится со всеми задачами благодаря:

Реляционная часть для финансов

JSONB для лендингов и документов

pgRouting (расширение) для маршрутов

Полнотекстовый поиск для документации

Но лучше так (полиглотная архитектура):

|Задача|	СУБД	|Почему|
|:-----|:-----------|-----:|
|Финансы|	PostgreSQL	|Надёжность, ACID|
|Лендинги/CRM|	MongoDB + PostgreSQL (JSONB)|	Гибкость|
|Документация|	Elasticsearch	|Быстрый поиск|
|Логистика|	Neo4j	|Скорость на графах|
|Связь между ними|	API/RabbitMQ	|Независимость|

### Решение

2.1. Пополнение баланса телефона (6 шагов)
Транзакция - это "всё или ничего". Либо деньги списались И зачислены, либо откат.

Пошагово:

|Шаг|	Действие|	Что происходит|	Если ошибка|

|1|	BEGIN TRANSACTION	|Система открывает |транзакцию  —|

## Задание 3. 
SQL vs NoSQL
3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL.

3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.

### Решение 
SQL vs NoSQL

3.1. Пять преимуществ SQL перед NoSQL

1	ACID (надёжность)	Транзакции либо выполняются полностью, либо откатываются. Деньги не теряются. NoSQL часто жертвует надёжностью ради скорости

2	Целостность данных	Можно настроить связи (один ко многим) и каскадное удаление. Удалил проект - удалились все его задачи автоматически

3	Сложные запросы (JOIN)	Можно соединить 5-10 таблиц одним запросом. NoSQL для этого нужны несколько запросов и сборка в коде

4	Стандартизация	SQL знают все. Перейти с MySQL на PostgreSQL легче, чем с MongoDB на Cassandra

5	Зрелость и инструменты	40+ лет развития. Миллионы инструментов для аналитики, отчётности, бэкапов. NoSQL моложе

### Главная идея: SQL - надёжный бухгалтер, который всё проверит и не ошибётся. NoSQL - быстрый курьер, который может что-то потерять, но доставит быстро.

3.1. Преимущества NewSQL*
NewSQL - это гибрид. Взял лучшее из SQL (ACID, транзакции, SQL-язык) и лучшее из NoSQL (горизонтальное масштабирование).

Преимущество -

Горизонтальное масштабирование	Как NoSQL - можно добавить 100 серверов, и БД сама распределит данные
ACID на распределённых данных	В отличие от NoSQL, транзакции работают даже на кластере из 1000 машин
Высокая производительность	Быстрее классического SQL за счёт распределения нагрузки
SQL-интерфейс	Не нужно учить новые языки запросов

## Примеры NewSQL: Google Spanner, CockroachDB, YugaByte DB, TiDB.
